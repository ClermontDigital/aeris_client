# AERIS Mobile — Android: create it properly + how to build/release

Practical, end-to-end guide for standing up the AERIS Android app on the
Google Play Console and cutting builds. No EAS — builds run on GitHub
Actions with native Gradle.

- **Package name (application id):** `com.aeris.erp` (from
  `mobile/app.json` → `android.package`). **Fixed at first Play upload —
  it can never change.**
- **Version name (marketing):** `mobile/app.json` → `expo.version` — the
  single source of truth for BOTH platforms (Info.plist / AndroidManifest
  are regenerated by `expo prebuild` every build, so never hand-edit them).
- **Version code (Android integer):** stamped from `GITHUB_RUN_NUMBER`
  (monotonic, unique per CI run).
- **Workflow:** `.github/workflows/android-build.yml`.

---

## 1. How the build works

Every run: `checkout → node 20 + JDK 17 + Android SDK 35 → npm ci →
patch jcenter() (Gradle 9 fix) → expo prebuild --platform android --clean
→ Gradle assembleRelease + bundleRelease → sign`.

Outputs:
- **APK** (`app-release.apk`) — direct install / sideload.
- **AAB** (`app-release.aab`) — the format Google Play requires.

Three ways to trigger it:

| Trigger | What you get |
|---|---|
| **Push a `mobile-v*` tag** | Signed APK + AAB **published to a GitHub Release** (`Aeris Mobile mobile-vX.Y.Z`), assets `aeris-X.Y.Z.apk` / `aeris-X.Y.Z.aab`. This is the canonical release path. |
| **Actions → "Android build (native Gradle)" → Run workflow** (`submit=false`) | Same signed APK + AAB, but as ephemeral **workflow artifacts** only (no Release). For ad-hoc sideloads. |
| **…Run workflow with `submit=true`** | Builds, then uploads the AAB to the Google Play **internal** track via `fastlane supply` (needs the Play service-account secret + an app that already exists on Play). |

## 2. One-time: create the app on Google Play

You only do this once. Until it's done, Play distribution isn't possible
(sideloading the APK works without any of this).

1. **Play Developer account** — one-time US$25 at
   <https://play.google.com/console>. Use the org's Google account.
2. **Create app** — Play Console → *Create app*:
   - App name: `AERIS`
   - Default language, App type = *App*, Free.
3. **Set the package name to `com.aeris.erp`** — it's fixed on the **first
   AAB upload**, so make sure the AAB you first upload was built from
   `app.json` with `android.package: com.aeris.erp` (it is). Get it wrong
   and you need a brand-new app record.
4. **Play App Signing (do this — it's the safety net):** on the first AAB
   upload, opt in to Play App Signing. Google then holds the *app signing
   key*; our CI upload keystore (below) is only the *upload key*. If the
   upload key is ever lost you can reset it — without Play App Signing, a
   lost keystore is terminal for the listing.
5. **Complete the required declarations** (Play blocks release until these
   are green): App content → Privacy policy, Data safety, Content rating,
   Target audience, Ads, Government apps, plus the store listing (short/full
   description, icon 512×512, feature graphic, screenshots).
6. **Create a testing track** — *Testing → Internal testing → Create
   release*, add tester emails, upload the AAB. Promote Internal →
   Closed → Production when ready.

## 3. One-time: CI signing secrets

The workflow signs with an *upload keystore* stored as repo secrets
(Settings → Secrets and variables → Actions). If they're not already set:

```bash
# Generate an upload keystore (keep the passwords + the .jks file SAFE —
# back it up to 1Password + an offline copy).
keytool -genkeypair -v -keystore upload.jks \
  -alias aeris-upload -keyalg RSA -keysize 2048 -validity 10000

# base64 it for the secret (macOS):
openssl base64 -A -in upload.jks | pbcopy
```

| Secret | Value |
|---|---|
| `ANDROID_KEYSTORE_BASE64` | base64 of `upload.jks` |
| `ANDROID_KEYSTORE_PASSWORD` | the store password |
| `ANDROID_KEY_ALIAS` | `aeris-upload` |
| `ANDROID_KEY_PASSWORD` | the key password |

> **Losing the keystore is only recoverable if Play App Signing is on**
> (step 2.4). Back it up.

### Optional — automated Play submission
For `submit=true` runs to push to the Play internal track:
1. Google Cloud → create a **service account**, enable the *Google Play
   Android Developer API*.
2. Play Console → *Users and permissions* → invite the service-account
   email → grant *Release to testing tracks* (at least).
3. Download its JSON key → `openssl base64 -A -in key.json | pbcopy` →
   add as secret `GOOGLE_PLAY_SERVICE_ACCOUNT_JSON`.

## 4. How to cut a release (the actual process)

1. **Bump the version** — edit `mobile/app.json` → `expo.version`
   (e.g. `1.5.1` → `1.5.2`). Keep `mobile/package.json` `version` in sync
   for tidiness. Commit. (Only `app.json` matters for the binary.)
2. **Tag it** on the commit you want to ship:
   ```bash
   git tag -a mobile-v1.5.2 -m "AERIS mobile 1.5.2 — <summary>"
   git push origin mobile-v1.5.2
   ```
   → the Android workflow builds and **publishes a GitHub Release**
   `Aeris Mobile mobile-v1.5.2` with `aeris-1.5.2.apk` + `aeris-1.5.2.aab`.
3. **Distribute:**
   - **Sideload (tablet):** download `aeris-1.5.2.apk` from the Release →
     `adb install -r aeris-1.5.2.apk` over USB (device in developer mode),
     or transfer the file and tap-install (enable *Install unknown apps*
     for the file source).
   - **Play Store:** download `aeris-1.5.2.aab` → Play Console → your track
     → *Create release* → upload. Or run the workflow with `submit=true` to
     auto-push the AAB to the internal track.
4. **iOS is separate** — push the `apple` branch → Xcode Cloud → TestFlight.
   (Same `app.json` version drives both, so bump once.)

### Quick commands

```bash
# Cut an Android Release from the current apple tip:
git tag -a mobile-v1.5.2 -m "…" && git push origin mobile-v1.5.2

# Or an ad-hoc artifact build (no Release) from a branch:
gh workflow run android-build.yml --ref apple -f profile=production -f submit=false

# Watch it + confirm the Release:
gh run watch "$(gh run list --workflow=android-build.yml -L1 --json databaseId -q '.[0].databaseId')" --exit-status
gh release view mobile-v1.5.2
```

## 5. Gotchas (already handled in CI, but know them)

- **`jcenter()` / Gradle 9** — `@react-native-cookies/cookies` still calls
  `jcenter()`, which Gradle 9 removed. A workflow step neutralises it after
  `npm ci`. If a dep bump reintroduces a `jcenter()` in another package, the
  build fails loudly at that line (extend the `find` glob in the step).
- **`versionCode` on a re-run** — it comes from `github.run_number`, which
  a GitHub "Re-run" reuses (only `run_attempt` bumps). Re-running a failed
  build can produce a *duplicate* versionCode, which Play rejects. If a run
  fails, **dispatch a fresh run / new tag** rather than "Re-run".
- **Never hand-edit `AndroidManifest.xml` / `Info.plist`** — `mobile/android`
  and `mobile/ios` are regenerated by `expo prebuild --clean` every build
  from `app.json`. Config changes go in `app.json`.
- **Marketing version already shipped** — once a version reaches the store,
  bump `app.json` before the next build (a repeat version is rejected on
  the store side).
