# AERIS Mobile — Cloud Build & Deploy Setup

How to build and ship the **AERIS** Expo app (`mobile/` in this monorepo) to the
App Store / TestFlight and Google Play, using the same pipeline the sibling
Clermont Digital Expo apps (`beats`, `plot_mobile`) already use:

- **EAS Build + EAS Submit** — Expo's cloud builders (iOS + Android).
- **Apple Xcode Cloud** — Apple's own cloud build, driven by a committed
  `mobile/ios/ci_scripts/ci_post_clone.sh` hook → archive → TestFlight.
- **GitHub Actions** — on-demand, credit-free Android `.aab` builds
  (`.github/workflows/android-build.yml`) + the existing release pipeline
  (`.github/workflows/build-release.yml`).

There are **two ways to connect** the cloud build. You can use **either or both**:

| Path | Who runs the build | Native project generated by | Best for |
|------|--------------------|-----------------------------|----------|
| **EAS** | Expo's servers (or `--local` on a GitHub runner) | EAS internally runs `expo prebuild` | Day-to-day iOS **and** Android, simplest signing |
| **Xcode Cloud** | Apple's servers | `mobile/ios/ci_scripts/ci_post_clone.sh` runs `expo prebuild` | iOS only; native/Xcode-side debugging, Apple-managed signing |

> AERIS is an **npm-workspaces monorepo** (`shared`, `mobile`, `client`). The app
> lives in `mobile/`. All commands below assume you start at the repo root unless
> a step says `cd mobile`.

4. **Google Play Console** (Android only) — an account with the `com.aeris.erp`
   app created and a service account JSON for API uploads (see §6).

## 2. EAS path — CLI

```bash
# Install the EAS CLI (or use `npx eas-cli@latest <cmd>` to avoid a global install)
npm i -g eas-cli

# Authenticate as a member of the clermont-digital org
eas login
eas whoami          # should print your user; org access via `owner` in app.json

cd mobile           # the Expo workspace
```

### 2a. Project link (already done — verify only)

`extra.eas.projectId` is **already populated** in `app.json`, so the project is
already linked to EAS. You should NOT need `eas init`. To verify:

```bash
eas project:info
```

If you are starting a *fresh* EAS project (e.g. a new bundle id), then:

```bash
eas init               # creates/links a project, writes extra.eas.projectId
                       # → paste the printed projectId into mobile/app.json
                       #   under  expo.extra.eas.projectId
```

### 2b. Credentials (signing)

EAS manages signing for you. First run is interactive; it creates and stores the
Apple distribution cert + provisioning profile and the Android keystore on EAS:

```bash
# iOS — Apple sign-in (Team 6SWY68AFK6); EAS creates/stores the dist cert + profile
eas credentials -p ios

# Android — generate the upload keystore EAS will keep and reuse
eas credentials -p android      # Android > production > set up a new keystore
```

### 2c. Build

```bash
# iOS production build on EAS servers (m-medium worker, channel "production")
eas build --profile production --platform ios

# Android production app-bundle (.aab) on EAS servers
eas build --profile production --platform android

# Both at once
eas build --profile production --platform all
```

Profiles available (in `mobile/eas.json`):
- `development` — dev client, simulator, `EXPO_PUBLIC_API_URL=http://localhost:8822`
- `preview` — internal distribution, `http://aeris.local:8000`
- `production` — store build, `m-medium`, android `app-bundle`, channel `production`,
  `EXPO_PUBLIC_API_URL=https://api.aeris.team`

### 2d. Submit to the stores

```bash
# iOS → App Store Connect / TestFlight. Reads submit.production.ios from eas.json
# (appleId shared@clermontdigital.com.au, appleTeamId 6SWY68AFK6, ascAppId 6762179693)
eas submit -p ios --latest          # or --id <buildId>

# Android → Google Play (see §6 first; needs a service-account JSON)
eas submit -p android --latest
```

> Tip: `eas build --profile production --platform ios --auto-submit` builds **and**
> submits in one step (this is what `build-release.yml` does on the `release` branch).

Everything above can be driven from https://expo.dev instead of the CLI:

## 4. App Store Connect — create / confirm the app record

An App Store Connect app record **already exists** for `com.aeris.erp`
(`ascAppId 6762179693`). If it does NOT (new bundle id), create it:

1. https://appstoreconnect.apple.com → **Apps → +** → **New App**.
2. Platform **iOS**, Name **AERIS**, primary language, Bundle ID
   **`com.aeris.erp`** (register it under **Certificates, IDs & Profiles** first
   if it isn't in the dropdown), SKU = any unique string (e.g. `aeris-erp`).
3. After creation, the **Apple ID** number shown on the app's **App Information**
   page IS the **`ascAppId`** — paste it into `mobile/eas.json` →
   `submit.production.ios.ascAppId`.
4. First TestFlight upload: run `eas build -p ios --profile production --auto-submit`
   (EAS) **or** let Xcode Cloud archive + deliver (§5). The build appears under
   **TestFlight** after Apple processing (~5–30 min). Add internal testers there.

## 5. Xcode Cloud path

Xcode Cloud is Apple's own CI. It clones the repo, runs our committed
**post-clone hook**, archives, signs with App-Store-Connect-managed certs, and
delivers to TestFlight — no EAS credits.

### How the hook is picked up

Apple runs `ci_scripts/` that sit **in the same directory as the Xcode project**.
AERIS's native project is at `mobile/ios/`, so the hook lives at:

```
mobile/ios/ci_scripts/ci_post_clone.sh   (committed, executable)
```

`mobile/.gitignore` ignores `ios/` wholesale (it's regenerated by `expo prebuild`),
so a negation rule in that `.gitignore` keeps **only** `ios/ci_scripts/` tracked.
The hook (monorepo-adapted):

1. Installs Node 20 + CocoaPods (+ cmake/ninja safety net), with retries for the
   flaky Xcode Cloud network sandbox.
2. Runs **`npm ci` at the repo root** with `--legacy-peer-deps --include-workspace-root`
   — the workspace install that resolves `@aeris/shared` for the mobile app.
3. `cd mobile`, runs `expo prebuild --platform ios --clean` to regenerate
   `mobile/ios/` (config plugins inject the camera / photo / Face ID Info.plist
   strings).
4. Pins `NODE_BINARY` for the xcodebuild "Bundle React Native" phase.
5. Runs `pod install`, producing `mobile/ios/AERIS.xcworkspace` — **the workflow
   must build the WORKSPACE, not the bare `.xcodeproj`**.

Signing: **automatic, App-Store-Connect-managed** (Xcode Cloud handles certs +
profiles for Team `6SWY68AFK6`). This is independent of EAS-managed credentials —
the two don't share signing assets.

### 5a. Connect via App Store Connect (web)

1. https://appstoreconnect.apple.com → your **AERIS** app → **Xcode Cloud** tab →
   **Get Started / Create Workflow**.
2. **Grant access to GitHub** → select this repository + the branch to build
   (e.g. `main` or `release`).
3. **Project / Workspace:** point it at `mobile/ios/AERIS.xcworkspace` and the
   **AERIS** scheme. (On the first connect the workspace doesn't exist in the repo
   yet — that's expected; the post-clone hook generates it. If the UI insists on a
   pre-existing workspace, do the initial connect from **Xcode** (§5b), which can
   read the locally-prebuilt project, then manage it on the web afterward.)
4. **Environment variables / secrets** (Workflow → Environment):
   - `EXPO_PUBLIC_API_URL` = `https://api.aeris.team` (the marketplace/relay base
     URL baked into the JS bundle; the hook defaults to this if unset).
   - `EXPO_TOKEN` — only needed if you later add an `eas update` publish step to
     the hook; the archive/TestFlight path itself does **not** require it.
   - Mark anything sensitive as **Secret**.
5. **Actions:** Archive → **Deploy to TestFlight (Internal Testing)**.
6. Save → run. Watch the **post-clone** log first; it must finish with
   `[ci_post_clone] done` before the archive starts.

### 5b. Connect via Xcode (desktop)

1. Generate the native project locally so Xcode has a workspace to open:
   ```bash
   cd mobile
   npx expo prebuild --platform ios --clean
   open ios/AERIS.xcworkspace
   ```
2. In Xcode: **Product → Xcode Cloud → Create Workflow**.
3. Select the **AERIS** app/scheme → connect the GitHub repo + branch (sign in to
   GitHub when prompted).
4. Set the same environment variables as §5a step 4.
5. Confirm signing is **Automatic** under the **AERIS** target → Signing &
   Capabilities (Team `6SWY68AFK6`).
6. Start the first build from Xcode or App Store Connect.

### EAS vs Xcode Cloud — how they complement

- **EAS** does iOS **and** Android, with EAS-managed signing and the simplest
  store-submit story. Use it as the default.
- **Xcode Cloud** is iOS-only, uses Apple-managed signing, and is handy when you
  want Apple-native build logs / to debug something in Xcode. It builds the *same*
  app from the *same* `expo prebuild`, so output parity is expected.
- Don't point both at auto-submitting the **same** build number to TestFlight in
  the same run, or Apple will reject the duplicate. Pick one to auto-deliver
  (typically EAS via `build-release.yml`); use the other for verification.

## 6. Android — Google Play

1. **Google Play Console** → create the app with package **`com.aeris.erp`**;
   complete enough of the first release that API uploads are accepted.
2. **Upload keystore:** `cd mobile && eas credentials -p android` → set up a new
   keystore (EAS stores it and reuses it for every build).
3. **Service account** (for automated submit): create a Google Cloud service
   account with Play Developer API access, download the JSON, grant it release
   permissions in Play Console.
4. **Submit options:**
   - **CLI:** `eas submit -p android --latest` (interactive path selection), or
   - **GitHub Actions** (`android-build.yml`): run the workflow with
     `submit: true` — it base64-decodes `GOOGLE_PLAY_SERVICE_ACCOUNT_JSON` and
     uploads to the **internal** track.
5. `mobile/eas.json` intentionally has **no** `submit.production.android` block yet
   (no service account is wired into the repo). The workflow passes `--path` + the
   decoded JSON directly, so submit works without it. Add the block later if you
   want `eas.json` to own the track/key config.

## 7. GitHub Actions — two workflows

| Workflow | Trigger | What it does |
|----------|---------|--------------|
| `build-release.yml` (existing) | push to `release` / `mobile-v*` tag | **Cloud** EAS builds: iOS `--auto-submit` to TestFlight, Android production. Spends EAS credits. The release pipeline. |
| `android-build.yml` (new, added here) | manual `workflow_dispatch` | **Local** EAS Android build on the runner (`--local`, **no** EAS credits), uploads the `.aab` artifact, optional Play submit. Mirrors beats/plot_mobile. |

Repository secrets to set (**Settings → Secrets and variables → Actions**):
- `EXPO_TOKEN` — Expo personal access token (create at expo.dev → Account →
  Access Tokens). Required by both workflows.
- `GOOGLE_PLAY_SERVICE_ACCOUNT_JSON` — base64 of the Play service-account JSON.
  Only needed when running `android-build.yml` with `submit: true`.
- (iOS auto-submit via `build-release.yml` uses the EAS-stored Apple credentials +
  `EXPO_TOKEN`; no extra Apple secret is required there.)
